# 2026-06-21 부분편입 조사·복원 — "누락"의 95%는 명명불일치 착시

## 질의
"손상 없는데 왜 데이터화 안 했나" — 입수 142 드라마 중 일부가 corpus에 1~몇 화만 보임.

## 결론: 거의 다 들어가 있었다 (명명불일치 착시)
- 1차 명명대조: 23 작품 부분편입 의심·339 누락처럼 보임.
- **내용 해시 dedup으로 검증**: 의심 아카이브 재추출 ~227편 중 **215편이 이미 corpus에 존재**(다른 work_id 이름으로). **진짜 신규는 12편뿐.**
- 거짓격차 원인: work_id 명명 불일치 — 태후NN↔태양의후예NN, 별순검S1NN, 범위파일(두번째프러포즈_1_8), 제목유실(신사의품격→"1부"~"20부"), 오타(눈시부시게), 적도↔적도의남자.

## 조치 (비파괴·추가)
- 중첩zip 재귀추출(태양의후예 태후1-16.zip 등) + 통합파일 파싱 + **내용중복 skip**.
- **진짜 신규 12편 추가**: corpus 2,339 → **2,351**. 12편 전수 임베딩 완료(append, todo 0). ChromaDB는 집 store_chroma 재빌드(additive).
- 기존 엔트리 무변경(개발자 검증상태 보존).

## 남은 진짜 이슈 = 손상/누락 아님, **명명 위생(cosmetic)**
- work_id 표기 불일치(태후/태양의후예, 제목유실 1부, 오타)로 *제목 기준 카운트·RAG·NextEp 페어링*이 어려움. 데이터(텍스트)는 보존. → 차기 work_id 정규화(rename) 권고, 단 임베딩/chroma id 영향이라 개발자 조율 후.

## 판정
**손상 0 · 미데이터화 0 · 진짜 누락 12편(복원완료).** 입수 대본은 사실상 전부 데이터화돼 있었고, "부분편입"은 명명불일치 착시였다. corpus=2,351 works.
