# SeqCard 층 데이터화 아키텍처 — v1 결정서

> 질문(사용자, 06-29): "대장금·도깨비 1화에서 작업한 방식·내용을 모든 한국드라마(+영화)에 적용하여, 거기서 **무엇을 스키마/청킹/JSON/임베딩/ChromaDB** 할지 정하는 것 아닌가."
> 목적: SeqCard(소제목+의도) 층을 **자율 기획·구성 학습 재료**로 데이터화. 산출 위상=분석/PoC(κ게이트 미통과, 학습 타깃 미승격).
> 전제: corpus_ko 본문 씬/청크/임베딩/ChromaDB는 **이미 존재**. SeqCard는 그 위에 얹는 **의도(설계도) 층**이며 코드·GPT 불가, Opus 독해 전용.

---

## 1. 결정 (ToT 3안 비교 후 선택)

| 안 | 내용 | 장점 | 단점/리스크 | 판정 |
|---|---|---|---|---|
| A 평면 확장 | 기존 씬 레코드에 title/intent/core 필드만 추가, 청킹·임베딩·chroma 그대로 | 최소 작업, 신규 스토어 0 | 회차·시리즈 롤업 없음→**아크 전이 학습 불가**, 의도 임베딩이 본문 임베딩과 섞임 | 기각(목적 상실) |
| **B 병렬 SeqCard 층** | 씬 본문 스토어는 그대로, **별도** 회차 jsonl+회차 메타 json+의도 전용 chroma 컬렉션+전이문법 테이블 추가 | 플래너 학습(P(다음\|현재)·아크 전이) 가능, 의도↔본문 분리, 설계 패턴 검색 | 스토어 수↑, scene_no 조인키 일관성 관리 필요 | **채택** |
| C 풀 그래프(NKG) | 카드마다 노드+타입엣지(precedes/pays_off/bookend) | 가장 풍부, 페이오프·북엔드 포착 | 카드 수 부족 단계서 과중, 스키마 churn, **미검증 라벨에 고비용 구조=위험** | 보류(B에 전방 포인터만) |

**선택 = B.** 근거: 사용자 목적(자율 기획 학습 재료)의 핵심은 **회차 간·씬 간 기능 전이**를 데이터로 만드는 것 → A는 이걸 못 만들고, C는 현 규모(실물 3편)·미검증(κ) 단계에 과투자. B는 학습에 필요한 최소충분 구조이며, C로 가는 길(엣지 필드 예약)을 막지 않음.

## 2. 스키마

### 2-1. 씬 카드 (scene-level, jsonl 1행=1씬) — 현행 확정
```
{work_id, scene_no, heading, title, intent_gist, core, core2, skin, by:"opus_reading"}
```
- `core`/`core2` = 16기능 택소노미(ESTABLISH·ORACLE·INTRO·BOND·CONFLICT·REVERSAL·LOSS·PUNISH·REVELATION·REUNION·RELIEF·ROMANCE·PERIL·RESCUE·DESIRE·HOOK).
- `skin` = 장소/시간(예 "특수부검실/낮"). 본문에서 분리해 둠(스킨-원장 분리 대비).
- **조인키 = (work_id, scene_no)** → 기존 씬 본문 레코드와 1:1.
- **C 전방 포인터(예약, 옵션 채움):** `payoff_to`(이 씬이 회수하는 앞 scene_no), `mirror`(북엔드 짝), `arc_tag`. 도깨비1화 PoC가 쓰던 why_link 계열. 지금은 비워도 되고, 보이면 채움.

### 2-2. 회차 메타 (episode-level, json 1개=1회차) — 신규
```
{work_id, scene_count, acts:[{name, scene_range, function, gist}],
 core_dist:{CORE:count}, engine:[엔진 노트 N개], episode_function}
```
- `episode_function` = 시리즈 안에서 이 회차의 역할(예 싸인1화="세계·인물 구축", 2화="갈등전개·진실전복"). **회차 간 전이 학습의 라벨**.
- md(사람용 설계도)와 1:1. md는 가독본, 이 json은 기계 소비본.

### 2-3. 시리즈 아크 (series-level, json 1개=1작품) — 신규(회차 누적 시 생성)
```
{series_id, n_episodes, episode_functions:[...], turning_points:[...], throughline}
```
- 전 회차 메타가 모이면 자동 집계. **P(회차기능 | 회차index, 총부수)** 학습 소스.

## 3. 청킹

| 단위 | 청크 | 용도 |
|---|---|---|
| 씬 카드 | 1씬=1청크 (이미 씬 입자) | 의도 임베딩의 기본 단위 |
| 시퀀스 카드(옵션) | core가 연속 동일/막 경계로 묶이는 2~5씬 | 미시플롯(비트) 검색 |
| 회차 카드 | 1회차=1청크(메타 요약문) | 회차 기능 검색·아크 전이 |

→ 본문 청킹(이미 듀얼: 씬+슬라이드)은 **건드리지 않음**. SeqCard는 의도 청킹을 **별도 축**으로 추가.

## 4. JSON / 파일 레이아웃
```
seqcards/{work_id}.seqcard.jsonl          # 2-1 씬 카드
seqcards/{work_id}.episode_meta.json      # 2-2 회차 메타 (신규, 다음 작업부터 동반 생성)
seqcards/{series_id}.series_arc.json      # 2-3 (시리즈 완주 시)
{work_id}.seqcard.md                      # 사람용 설계도(가독본)
```
- 회차 단위 파일 유지(증분·재처리 쉬움). 작업 = 정확분할 + 카드 채움 + 회차메타를 **한 패스**로.

## 5. 임베딩
- 모델 = **text-embedding-3-small**(기존 본문과 동일 모델, 허용됨). GPT 텍스트 생성 아님.
- 임베딩 대상 = `title + " " + intent_gist`(의도 의미). **본문 text 임베딩과 분리된 "의도 공간"**.
- 회차 메타는 `engine`+`episode_function` 합성문을 임베딩(회차 검색용).
- 왜 분리: 본문 임베딩은 "무슨 말이 오갔나", 의도 임베딩은 "이 씬이 **무슨 일을 하나**". 플래너가 필요로 하는 건 후자.

## 6. ChromaDB
- **신규 컬렉션 `seqcard_intent`** (본문 컬렉션과 별개).
- 메타데이터: `work_id, scene_no, core, core2, skin, act, episode_function, by`.
- 가능 질의: "core=REVELATION이며 회차말 HOOK으로 닫는 씬", "사극의 ESTABLISH 도입 패턴", "추격 시퀀스(PERIL 연속) 사례".
- 본문 ChromaDB와 동일하게 **emb_cache(npy)가 진본**, chroma는 파생물(재빌드 가능).

## 7. 학습 산출물(왜 이렇게 데이터화하나)
1. **전이문법 prior** `P(core_next | core_cur, 회차내 위치, 장르)` — 씬 카드 시퀀스에서 직접 집계. 플래너의 미시플롯 골격 prior.
2. **회차 기능 전이** `P(회차기능 | 회차index, 총부수)` — 회차 메타에서. 거시 아크 prior.
3. **설계 패턴 검색(RAG)** — 생성 시 "지금 만들 씬과 같은 기능을 한 실제 사례" 회수 컨텍스트.
→ 세 가지가 사용자가 말한 "스스로 시나리오를 기획·구성·마이크로플롯 구성"의 학습 기질(substrate). κ게이트 통과 후 prior로 주입.

## 8. 적용 순서(전 작품)
- 하이브리드 유지: A(결함 재분할 필요분: 싸인 등 fallback_block) + B(κ골드: 도깨비·미생·시그널·태양의후예) 세션마다 혼합.
- **시리즈 완주 우선 권고:** 싸인은 결함군(A)이자 완질 시리즈 → **싸인 전19부 완주**가 회차 간 아크 전이를 처음으로 데이터화하는 최적 첫 사슬(하이브리드 스윗스팟).
- 회차 메타 json은 **다음 작업분(싸인_03~)부터 동반 생성**. 기존 3편(대장금01·싸인01·02)은 md에 엔진/막 구조가 이미 있어 소급 생성 저비용.

## 9. 자기 점검(약점)
- κ 미검증 라벨에 임베딩·컬렉션을 구축 = **분석/PoC 위상 고정**, prior 주입은 κ≥0.6 후. (구조는 미리 만들되 학습 타깃 승격은 금지 — 일관 유지.)
- core 단일 분류의 자의성 → core2 보조 + (work_id,scene_no) 보존으로 재라벨 가능(파괴적 아님).
- 측정 캐비엇: 하이픈/시간코드 포맷 결함군은 분할 배수 보수적으로(원본 1:1 검증 메모 §캐비엇 준수).
