# Phase E 가치검증 — PROXY-MVE 실행 결과 (2026-06-02)

## 0. 무엇을 실제로 돌렸나 (정직한 범위)
- **실행됨**: 동일 프롬프트 4씬 × 2팔(A=순수 LLM, B=구조주입 LLM) 실생성 + 블라인드 LLM 판정 + 객관지표. 실제 API 호출.
- **모델**: 생성·판정 모두 gemini-2.5-flash (Anthropic·OpenAI 키는 잔액/쿼터 소진으로 사용 불가 → Gemini로 실행).
- **PROXY 표기**: arm B는 실제 Literary OS 파이프라인(NKG·실코퍼스·공식 엔진)이 아니라 **구조를 프롬프트로 근사**한 것. 판정자도 인간 작가가 아니라 LLM. 따라서 본 결과는 **방향성 신호**이지 G_VALUE_PROOF 자체가 아님.

## 2. 해석 (과장 없이)
1. **무승부.** 구조를 프롬프트로 주입한 PROXY는 순수 프롬프트를 명확히 이기지 못했다. 총점은 A가 근소 우위지만 N=4로 통계적 의미 없음.
2. **교란변수 발견 — 길이 불균형.** B가 체계적으로 짧음(557 vs 874자). 긴 구조 지시가 출력 예산을 잠식했거나 제약이 장면을 압축시킴. immersion에서 B가 가장 낮은 건 길이 탓일 개연성. **실험 설계 결함이며 본 실험에서 출력 길이를 통제해야 함.**
3. **가설은 반증되지 않음.** 이건 프록시(얕은 구조)일 뿐. Literary OS의 핵심 주장은 '프롬프트 주입보다 깊은 NKG·공식 엔진'이므로, 본 결과는 오히려 **"구조를 얕게 흉내내는 것만으로는 이득이 없다 → 깊은 파이프라인의 실제 우위를 반드시 실측해야 한다"**를 보여준다.

## 3. 하니스는 작동한다 (실행 증명)
`harness_one.py`가 실생성→블라인드 판정→점수 집계를 끝까지 수행. experiments/value_proof 하니스의 골격이 실제 동작함을 입증.

## 4. 본 실험(G_VALUE_PROOF)로 가기 위한 교정
1. **출력 길이 통제**(양 팔 동일 목표 길이·동일 토큰예산, 구조 지시는 system에 분리).
2. **arm B = 실제 파이프라인**(NKG·공식·RAG), 프롬프트 프록시 아님.
3. **판정자 = 인간 작가 5+**(LLM 판정은 보조·사전 스크리닝만).
4. **N 확대 + 사전등록**(B 선호 ≥60%, p<0.05) — 현재는 과소표본.
5. **공식-Critic vs 블랙박스 보상모델 ablation**(검증 2) 추가.

## 5. 결론
실행 이행 완료. 현 시점 신호 = **무승부(교란 있음)**. 이는 실패가 아니라 **"존재가치는 얕은 프록시가 아닌 깊은 파이프라인 + 인간 평가 + 길이통제 + 충분한 N에서만 증명된다"**는 점을 정량적으로 확인한 것. 다음은 길이통제판 재실행 또는 실제 LLM-1 파이프라인 연결 후 본 실험.
