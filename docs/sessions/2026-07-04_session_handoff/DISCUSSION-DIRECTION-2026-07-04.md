## 2. 오늘의 개념적 발견 (이게 더 중요)

### 2.1 ★상위 3층이 비어 있었다 — 자기점검으로 드러남
"SeqCard"라는 이름과 달리 실제 산출은 **씬 카드뿐**이었다. 사용자의 4단 이해 중 아래 하나만 존재:

| 층 | 이해 내용("~이다") | 현 상태 |
|---|---|---|
| 전체 시놉시스 | 이 작품은 이런 이야기·이것을 말한다 | ❌ 없음 (series_arc = 통계뿐) |
| 회차 시놉시스 | 이 회는 이런 이야기다 | ❌ 없음 (episode_meta = 기능라벨뿐) |
| 시퀀스 의도 | 이 묶음은 왜·무엇을 하나 | ❌ 없음 (개수만 측정) |
| 씬 의도 | 이 씬은 무엇을 하나 | ✅ 30작 전수 존재 |

→ **하향 사슬(전체→회차→시퀀스→씬)이 끊겨 있어, top-down 자율 기획의 근거가 없다.** 씬만 아무리 모아도 "전체가 이러니 이 회가 이렇고 이 시퀀스가 이걸 해서 이 씬이 나온다"를 만들 수 없다. = **1차 목표의 핵심 공백.**

### 2.2 방향 — 순서를 뒤집는다 (역채움)
이미 씬은 다 있으니, 같은 30작에 대해 **씬 위의 3층을 거꾸로 채워 올려** 4층이 다 연결된 **수직 완결 사례 1편**을 먼저 만든다(후보: 비밀의숲_01, 이미 씬 슬라이스 검증에 사용).
- **단, 4층 역채움의 상세 설계는 추후 멀티에이전트와 제대로 심의**(사용자 지시로 defer).

### 2.3 드라마 분석은 계속 — 종착은 180편 전량
개발자 입수 ~180편을 **결국 전부 분석**한다. 그 분석 과정 자체가 우리의 학습이다("작가들이 실제로 어떻게 위에서 아래로 구성했나"를 데이터로 축적 = 상위 3층·거시 판단의 근거). 현재 30작은 그 여정의 초입.

### 2.4 ★수정 전파 = Git-Nexus 원리 (사용자 제안 + 합의)
2차 목표의 human-in-loop 완성("골격 → 사용자와 가감수정 → 완성")을 위한 수정 전파 엔진:
- **채택 (Git의 구조):** 노드(시놉시스·회차·시퀀스·씬)를 **의존성 DAG**로. 사용자 수정 = **커밋/diff**. 이력·롤백·분기(대안 버전)·영향범위 추적으로 **하위 의존 노드만 골라 재생성**.
- **교체 (Git의 merge):** Git은 텍스트를 줄 단위로 기계 병합 → 이야기엔 부적합. "3화에서 인물 사망"을 고치면 4화 이후가 **뜻으로** 재구성돼야 함. → 3-way 텍스트 머지 자리에 **LLM 기반 의미 재생성**을 놓는다.
- **한 줄:** Git의 그래프·이력·의존성은 쓰고, **병합 알고리즘만 의미 전파로 교체.**
