## 1. 졸업(SP-E.10 / v14.0.0)의 의미 — 쉽게, 그리고 자세히

### 1.1 한 문장
**우리 8B 모델이 "쌍대 비평가(Critic)"로서 글의 좋고 나쁨을 판단하는 능력을, 공식이 아니라 스스로 학습으로 5라운드 연속 진짜로 향상시켰음을 실측 증거로 증명한 사건.** 이게 LLM-1의 정의이고, 그래서 "졸업"이다.

### 1.2 용어를 일상어로
- **loop-C (자기개선 루프)**: 모델에게 "잘 쓴 글(chosen)"과 "못 쓴 글(rejected)" 쌍을 보여주고, 잘 쓴 쪽을 더 좋아하도록 미세조정(DPO)한 뒤, 정말 좋아하게 됐는지 다시 시험 보는 사이클. 한 바퀴 = 1라운드.
- **W (win-rate, 승률)**: 시험에서 모델이 "잘 쓴 글"을 골라낸 비율. 0.5면 동전던지기(못 가림), 1.0이면 완벽히 가림. **per-token W**=글 길이가 길어서 점수가 뻥튀기되는 착시를 제거한, 길이중립 승률. (이전 v1·v2 실패의 주범이 이 착시였음.)
- **adopt / rollback**: 한 라운드 학습이 진짜 향상이면 채택(adopt), 망가졌으면(드리프트·과적합) 직전 상태로 되돌림(rollback).
- **졸업 불변식(graduation_invariant)**: "말미 5라운드가 **연속 adopt** + 누적 시험쌍 ≥250 + 매 라운드 승률 신뢰구간 하한 >0.5 + 길이규칙률 ≤0.6 + 구조게이트(c3) 통과." 이 6개가 동시에 0위반이어야 졸업.

### 1.3 실측이 말하는 것 (round_records_v3.json, 집 RTX 4070)

| 라운드 | 학습 전 W0 | 학습 후 W1 | 신뢰구간 하한 | 길이규칙률 | c3 | 판정 |
|---|---|---|---|---|---|---|
| 1 | 0.580 | 0.600 | 0.539 | 0.0 | ✓ | adopt |
| 2 | 0.596 | 0.620 | 0.560 | 0.0 | ✓ | adopt |
| 3 | 0.616 | 0.644 | 0.585 | 0.0 | ✓ | adopt |
| 4 | 0.640 | 0.708 | 0.652 | 0.0 | ✓ | adopt |
| 5 | 0.712 | 0.808 | 0.759 | 0.0 | ✓ | adopt |

읽는 법:
- **W가 0.580→0.808로 한 칸씩 꾸준히 올랐다.** 한 라운드에 천장(1.0)으로 튀지 않고 매번 "조금씩" 이겼다 = **벼락치기 암기가 아니라 진짜 실력이 단계적으로 늘었다**는 증거. (이전 v2 실패는 2라운드 만에 W=1.0으로 포화 → 더 배울 게 없어 연속 adopt 구조가 깨졌던 것.)
- **길이규칙률 = 전 라운드 0.0.** 길이로 점수 따먹는 꼼수가 한 번도 없었다 = 내용으로 이겼다.
- **신뢰구간 하한이 매번 0.5 초과.** 운(우연)으로 이긴 게 아니라 통계적으로 유의했다.
- **6/6 불변식 충족, violations=[], exit_version=v14.0.0.**

### 1.4 이게 왜 "사다리 한 칸"인가
이 졸업으로 **판단의 주체가 공식에서 LLM으로 넘어갈 수 있음**이 처음으로 수치 증명됐다. 지금까지(Phase A~D)는 "이 글이 좋은가?"를 전부 공식이 채점했다. 이제 모델 자신이 쌍대 노트(둘 중 뭐가 낫고 왜인지)를 줄 수 있다 = **LLM-1 = 비평가 자격 취득**. 다음 칸(LLM-2)은 "판단"을 넘어 "생성 주력"으로 가는 단계다.

### 1.5 정직한 경계 (조기 승리선언 금지 — 개발자가 명시)
- 졸업한 능력은 **show/tell 한 축**(보여주기 vs 설명하기, 글쓰기 기교 한 칸)이다. **거시 기획(16/24부작 전체 설계)은 여전히 빈칸** = LLM-2의 일.
- rejected가 "AI가 쓴 평이한 글"이었다. **인간 명작과의 직접 대조는 최종시험으로 남아있다.**
- floor(공식 바닥)는 "측정 쉬운 축으로 쏠리는 방향"까지는 못 막는다 → 차기 단계에서 양성 평가축 + 축간 균형 메타게이트가 필요(3-전문가팀 경고).

**요약: 진짜지만 좁은 승리.** 한 칸은 확실히 올랐고, 그 한 칸이 "공식→LLM 판단 이양"의 가능성을 증명했다.

## 2. Synopsis Assembler 계약 — 아주 자세히

배경: 최상위 파이프라인 `DramaEpisodeGenerator.generate_series`는 지금 `DramaSeriesConfig`의 **logline·characters를 사람이 손으로 박은 상수**로 받아 화 루프만 돈다. 즉 "작품 컨셉→인물→아크→복선"을 채우는 **자율 머리가 없다**(코드에 `Synopsis/Logline/StoryBible` 클래스 0건). Synopsis Assembler = 그 빈 머리.

### 2.1 입력 — `SynopsisRequest` (씨앗)
사람이 주는 최소 씨앗. **비우면 모델이 채우고(자율도↑), 채우면 작가가 고정(통제↑)** — 이 한 장치가 "통제권 이양"의 손잡이다.

| 필드 | 의미 | 비우면 |
|---|---|---|
| `genre` (필수) | 주장르(+혼합 비중; K드라마는 코믹/멜로 상시혼합) | 에러 |
| `premise` | 한 줄 전제·소재 | 로그라인 역생성이 보강 |
| `logline` | 작가가 이미 가진 로그라인 | 생성모델이 발산 |
| `constraints` | `{episodes, runtime_min, tone, target}` 작가 고정값 | 기본값(16부·60분 등) |
| `seed_characters` | 작가가 미리 박는 인물 | 생성모델이 캐스팅 |

### 2.2 출력 — `SynopsisDraft` 7요소 (+개발자 추가 1급)
시놉시스 한 장 = 아래 슬롯. **각 슬롯이 §2.3 기관의 입력으로 직결**되는 게 핵심(그래서 "조립"이다).

| # | 요소 | 무엇 | 생산 |
|---|---|---|---|
| ① | **로그라인** | 작품을 한 줄로 | 생성모델(역생성) |
| ② | **주제(aboutness)** | 이 작품이 결국 무엇에 관한가 | 생성모델(①과 동시) |
| ③ | **인물 세트** | 주인공/반대/주변 + 등장 근거 | 생성모델 → birth_gate 검증 |
| ④ | **중심 갈등** | 이야기를 미는 핵심 대립 | 생성모델 + conflict_collision 점검 |
| ⑤ | **거시 아크** | 4막(기25/승35/전25/결15)·텐션곡선·복선예산 | SeriesArcPlanner.plan() |
| ⑥ | **화별 비트** | 회차별 미시플롯 수 K·감정·갈등·회말 훅 | EpisodePlanner.plan() ×N |
| ⑦ | **복선 스케줄** | 어느 화에 심고 어느 화에 터뜨릴지 | PayoffScheduler.generate_schedule() |
| ★ | **causal_spine**(개발자 추가, 1급) | 사건 간 "진짜 인과"의 DAG | LLM 인과분류 패스(아래 §4 참조) |
| (메타) | tone_genre_profile / track_axis(시간선형·진영형) / series_config / provenance | 톤·작품군축·진입설정·생산이력 | 조립 시 |

### 2.3 동일 호출 그래프 (제 IO-v1 ≡ 개발자 설계, 독립 수렴)
```
SynopsisRequest(씨앗)
   │
   ▼ [로그라인 역생성]  ──► 로그라인①·주제②
   ▼ SeriesArcPlanner.plan()        ──► CausalPlotGraph = 거시 아크⑤ (4막·텐션·복선예산)
   ▼ CausalPlotGraph                ──► 사건 인과·복선 후보 엣지
   ▼ EpisodePlanner.plan(series_config, ep, narrative_state) ×N ──► 화별 비트⑥
   ▼ PayoffScheduler.generate_schedule(residue_ids, strategy)   ──► 복선 스케줄⑦
   ▼ 인물엔진(birth_gate/ledger/intent_agent)                   ──► 인물 세트③
   ▼ SynopsisDraft 조립 → generate_series(draft.series_config)
```
- **SeriesArcPlanner**: 16부 4막 골격·S자 텐션·복선예산을 자동 산정. 입력=총화수·제목.
- **CausalPlotGraph**: 에피소드 간 인과·복선 엣지 그래프. (★주의: `infer_causal_edges`는 **패턴 기반·LLM 0회** — §4의 한계 근거.)
- **EpisodePlanner**: 화별 미시플롯 수 K 등 9변수를 결정론으로 산정 → EpisodePlan.
- **PayoffScheduler**: residue(심어둔 복선)를 16부에 분배. ★`get_episode_brief(r)`이 핵심(§5).
- **인물엔진**: 등장 판정(birth_gate)·원장(ledger)·의도(intent_agent).

### 2.4 생성모델 인터페이스 추상화 (두 문서 공통 합의)
호출 그래프에서 **"창작"을 하는 부분(로그라인·인물·갈등 생성)을 구체 모델에 직접 묶지 않고 인터페이스 뒤에 숨긴다.** 이유: 지금 그 모델은 **갓 졸업한 LLM-1(8B)** 이라 초안 품질이 미보장. LLM-2 졸업 후엔 프론티어 모델로 갈아끼워야 한다. 인터페이스로 추상화하면 **모델만 교체, 계약은 불변.** (개발자 문서는 "인터페이스로 추상화"라고 선언만, 제 IO-v1은 이를 구체 3구현으로 — §6.3.)

### 2.5 공식 = floor, 발동률 0 수렴
- 공식은 **폐기되지 않는다.** "안전 바닥(floor)"으로 남아, 생성모델이 헛소리를 하면 최소 기준을 강제한다(구조게이트 c3, 트레이너 BANDS, drse, emotion 밴드).
- 단, **발동 빈도가 0으로 수렴하는 것**이 진보의 지표다 = "공식이 나설 일이 점점 없어진다" = 모델이 알아서 잘한다. LLM-2~3로 갈수록 floor 발동률↓ + 자가평가 일치도(κ)↑.
- 즉 "공식 제거"가 아니라 "**공식이 침묵하는 빈도 측정**"이 이양의 정의다.

### 2.6 빈칸 3종 (LLM-2가 새로 만들 것)
1. **Synopsis Assembler 본체** — 위 머리. 설계만 있고 구현 0(클래스 부재 확인).
2. **로그라인/주제 역생성** — 전제→로그라인·aboutness. 생성모델 + 명작 로그라인 코퍼스 대조. (INDEX 의제 O7)
3. **수정 전파 엔진** — 작가가 한 부분(노드)을 고치면 인과로 엮인 뒤 이야기를 다시 짜는 엔진. PayoffScheduler.rebalance가 기반이나 완전판은 빈칸. (INDEX 의제 O8)

## 3. (참고) 개발자 설계가 더 나아간 점 — causal_spine
개발자 문서는 제 IO-v1에 없던 **인과 척추(causal_spine)를 1급 산출로 승격**했다. 근거: (b)파일럿에서 룰 인과 vs 진짜 인과 일치 **0.56**, 위음성 8/32(인물이 안 겹쳐도 사물·정보·사건으로 잇는 인과를 룰이 놓침). → 패턴 기반 후보 엣지 위에 **LLM 강제력 판정 패스**를 얹어 causal_spine=DAG를 1급으로, 인물 분포는 부산물로 강등. 또 **`get_episode_brief`**(§5)로 "8B가 16부를 안 봐도 되는" 메커니즘을 제시 — 이건 제 IO-v1 자기점검의 미해결(narrative_state 초기화 책임)에 대한 답이다.

## 4. 제 IO-v1 문서가 더 나아간 점 (병합 가치) — 자세히

개발자 설계가 "무엇을 만들지(causal_spine·요소)"에 강하다면, 제 IO-v1은 "**어떻게 안전하게 조립·교체할지(아키텍처 계약)**"에 강하다. 네 가지.

### 4.1 결속 방식 ToT 3안 비교 + 중앙상태(B) 제거 논증
"Assembler가 기관들과 어떻게 묶이는가"를 3안으로 동시 평가:

| 안 | 구조 | 판정 |
|---|---|---|
| A 얇은 오케스트레이터 | 순서대로 직접 호출, 상태 없음 | 최소코드지만 모델 교체 시 곳곳 침투 |
| **B 중앙 StoryBible 상태** | 모든 기관이 공유 상태를 읽고 씀 | ★**제거** |
| **C 계약우선 포트/어댑터** | 안정계약 + 어댑터 + GenerativePort | ★**채택** |

**B 제거 논증(중요):** 80+ 모듈에 공유 상태를 박으면 작은 변경마다 전수 회귀가 터지고, 메인 트랙의 "점진 완화·트랙 분리" 철학에 정면 위배된다. **수정 전파(빈칸③)는 별도 엔진으로 분리**해야지, 계약 단계에서 중앙상태로 끌고 들어오면 안 된다. → 이 논증이 없으면 누군가 "story_bible을 공유 상태로 만들자"는 자연스러운 유혹에 빠지기 쉬움(개발자 ORCHESTRATOR 문서의 story_bible 개념과 충돌 가능 — 병합 시 점검 지점).

### 4.2 Stage 어댑터층 (기관 무수정)
각 기관(SeriesArcPlanner·EpisodePlanner·PayoffScheduler…)을 **균일한 `Stage` 인터페이스로 감싸는 얇은 어댑터**를 둔다. 어댑터는 v794 실측 시그니처를 **그대로** 호출만 한다(기관 코드 0줄 수정). 효과: Assembler는 `Stage` 리스트를 순서대로 실행하는 단순 DAG 실행기가 되고, 단계 추가·교체·재배열이 자유로워진다. causal_spine 패스도 "S4와 S5 사이의 새 Stage" 하나로 깨끗이 삽입된다(= §3 개발자 설계와 무봉합 결합점).

### 4.3 GenerativePort 3구현 = 판단 이양 축을 "계약"으로 구현
생성모델 인터페이스(`GenerativePort`)의 **구체 구현 3개**를 못 박아, 추상 선언을 실행 가능한 교체 계획으로 만든다:

| 구현 | 시점 | 역할 |
|---|---|---|
| `FormulaFallbackPort` | 상시 | 공식 baseline. Port 실패·저신뢰 시 발동. **발동률→0 수렴이 졸업지표**(§2.5와 직결) |
| `LLM1Port` | 현재(졸업 직후) | 8B. 초안 품질 미보장(절대 미선언) |
| `FrontierPort` | LLM-2 졸업 후 | 프론티어 LLM. **계약 무변경 스왑** |

→ 이게 핵심: "공식 floor 잔존 + 발동률 0 수렴 + 8B→프론티어 격상"이라는 **판단 이양 축의 합의가, 인터페이스 3구현이라는 코드 계약으로 1:1 번역**된다. 측정 훅(provenance에 생산자 기록)으로 발동률을 상시 계측.

### 4.4 8항 DoD 체크리스트 (무GPU 선검증 경로)
구현 모드 인계용 완료기준 8항. 핵심 순서: **`FormulaFallbackPort`로 먼저 무GPU E2E 1작품 조립을 통과**시켜 배관(7단계 호출 그래프)이 도는지 결정론으로 증명 → 그 다음 `LLM1Port`를 꽂아 초안 비교. 효과: GPU·API 비용 0으로 머리의 골격을 먼저 세우고, 모델은 나중에 끼운다 = 회사세션(무GPU) 범위에서 LLM-2 첫 실체의 절반을 만들 수 있음.

## 5. (보강) get_episode_brief — 8B 장거리 한계의 답
개발자 CAPACITY-DIVISION의 메커니즘. PayoffScheduler가 16부 전체 복선 원장을 **외부 메모리**로 들고, 회차 r에 `get_episode_brief(r)`="이 화에 심을 복선/터뜨릴 복선/예산"만 반환. → 8B는 **그 화 brief + 직전 상태 요약**(바운디드)만 받아 그 화만 쓴다. 장거리 일관성(1화 복선→14화 회수)은 결정론 엔진이 외부에서 보장. = "8B로 16/24부작?"의 답 = **8B에게 거시를 안 시킨다**(거시=결정론 외부메모리+프론티어 판정, 8B=바운디드 로컬 산문). 이 메커니즘은 §4.2 Stage 어댑터에서 "PayoffScheduler Stage가 회차 루프에 brief를 공급"으로 자연 수용된다.

## 6. 권고 — 병합(중복 push 아님)

### 6.1 왜 병합인가
제 IO-v1은 허브 미push 상태다. 개발자 정본 = 허브 `DESIGN-LLM2-SYNOPSIS-ASSEMBLER-v1`. 둘은 독립 작성인데도 동일 결론에 수렴했다(=상호검증). 따라서 **같은 문서를 또 올리는 건 잡음**이고, 옳은 건 **제 아키텍처 계약을 개발자 §3 causal_spine 설계에 얹는 병합**이다:
- 개발자 §3(causal_spine·요소·get_episode_brief) = **무엇을** 만드는가 → 유지.
- 제 IO-v1(ToT 결속·Stage 어댑터·GenerativePort 3구현·DoD) = **어떻게** 안전 조립·교체하는가 → 개발자 §3 위에 "구현 계약" 절로 삽입.
- 결합점: causal_spine 패스 = §4.2 Stage 하나, GenerativePort = §2.4 추상화의 구체화, get_episode_brief = §4.2 PayoffScheduler Stage가 공급.

### 6.2 다음 의제 (택1)
- **① 병합 실행**: 개발자 SYNOPSIS-ASSEMBLER-v1에 "구현 계약(Stage/GenerativePort/DoD)" 절을 증보하여 v2로 push. (본 문서가 그 초안.)
- **② O9 판단 이양 PoC**: 5축 중 1축 또는 플로어 1종을 8B→프론티어로 바꿔 공식 대비 일치도(κ)·정확도 측정 → 발동률 0 수렴 경로의 첫 수치. (Phase F 선행, INDEX O9.)

권고: **①을 먼저**(설계 정합부터 닫고), ②는 Phase F 진입 신호로.
