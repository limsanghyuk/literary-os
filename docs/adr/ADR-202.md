# ADR-202: Disaster Recovery Backup Strategy — DRBackupManager (V741)

| 항목 | 내용 |
|------|------|
| 상태 | Accepted |
| 결정자 | CSA · CSPE |
| 날짜 | 2026-05-29 |
| 관련 | ADR-203, ADR-204, D-M-11, RS-09 |

## 맥락

ADR-192 §RS-09에서 확정한 DR RPO 위험:
"backup_interval_seconds > 3600 또는 복원 로직 오류 시 RPO 1h 위반".

Literary OS v13.0.0의 엔터프라이즈 SLA는 RPO ≤ 1h를 의무 조건으로 포함하며,
이를 코드 레벨에서 강제하는 `DRBackupManager`가 필요하다.

## 결정

`literary_system/disaster_recovery/backup_manager.py`에 `DRBackupManager` 구현.

### 핵심 설계 결정

1. **RPO 불변식**: 생성자에서 `backup_interval_seconds > 3600` 시 즉시 `ValueError`
   ```python
   assert backup_interval_seconds <= 3600  # 1h = 3600s
   ```

2. **SHA-256 체크섬**: 모든 백업에 `hashlib.sha256(data).hexdigest()` 포함
   - 복원 시 무결성 검증 가능

3. **자동 프루닝**: `max_retained_backups` 초과 시 가장 오래된 항목 제거
   - 기본값 24 (1h 간격 기준 24시간 보존)

4. **테넌트 격리**: `_records: Dict[str, List[BackupRecord]]` — 테넌트별 독립 저장

5. **RPO 준수 확인**: `rpo_compliant(tenant_id, reference_time)` 메서드
   - `age = reference_time - latest_backup.created_at`
   - `age <= backup_interval_seconds` → compliant

## 결과

- Phase D 종료 전 DR 인프라 기반 확보
- G91 게이트(ADR-203)에서 이 모듈을 직접 검증
- Phase E에서 실제 스토리지 연결 시 인터페이스 유지

## 기각된 대안

| 대안 | 이유 |
|------|------|
| 외부 백업 라이브러리 | CI 의존성 증가, 인터페이스 제어 불가 |
| RPO를 런타임 경고로만 처리 | SLA 위반 자동 차단 불가 |
| 테넌트별 파일 시스템 격리 | Phase D에서 실제 스토리지 불필요 |
