# ADR-173: PluginRegistry — 중앙 플러그인 레지스트리 설계

**상태**: Accepted  
**날짜**: 2026-05-28  
**버전**: V712  
**결정자**: SP-D.3 설계팀

---

## 맥락

PluginManifest(V711)와 PluginLoader(V711)만으로는 다수 플러그인의 생명주기·조회·훅 관리가 불가능하다.
서비스 레이어에서 "현재 등록된 플러그인 목록", "태그별 필터링", "로드 상태 추적"을 일관되게 처리할
단일 중앙 저장소가 필요하다.

## 결정

`literary_system/plugins/plugin_registry.py`에 `PluginRegistry` 클래스를 구현한다.

### 핵심 설계 원칙

1. **단일 진실 공급원**: 모든 PluginManifest는 PluginRegistry를 통해서만 등록/해제된다.
2. **옵트인 자동로드**: `auto_load=True`일 때만 `register()` 시점에 PluginLoader.load() 호출.
3. **훅 시스템**: `on_register` / `on_unregister` 훅 목록 — 콜백이 실패해도 레지스트리 동작은 보장.
4. **덮어쓰기 명시성**: `overwrite=True`를 명시해야만 동일 plugin_id 재등록 허용.
5. **조회 다양성**: tag, permission, loaded 상태로 필터링 가능.

### `RegistryEntry` 구조

```python
@dataclass
class RegistryEntry:
    manifest:    PluginManifest
    load_result: Optional[PluginLoadResult] = None
```

### `PluginRegistry` 공개 API

| 메서드 | 설명 |
|--------|------|
| `register(manifest, overwrite=False)` | 등록 (+auto_load 시 즉시 로드) |
| `unregister(plugin_id)` | 등록 해제 (+로드된 경우 unload) |
| `get(plugin_id)` | 단건 조회 |
| `list_all()` | 전체 RegistryEntry 목록 |
| `query_by_tag(tag)` | 태그 필터 |
| `query_by_permission(permission)` | 권한 필터 |
| `query_loaded()` | 로드 완료 항목만 |
| `on_register(hook)` | 등록 훅 추가 |
| `on_unregister(hook)` | 해제 훅 추가 |
| `count` (property) | 등록 수 |
| `loader` (property) | 내부 PluginLoader |

## 대안

- **글로벌 싱글턴 레지스트리**: 테스트 격리 불가, 채택 안 함.
- **DB 백엔드**: V712 범위를 벗어남. 추후 ADR로 결정.

## 결과

- PluginRegistry가 SP-D.3 플러그인 레이어의 진입점이 됨.
- V713 이후 Whitelist/Sandbox는 PluginRegistry를 통해 플러그인에 접근.
- G87 Gate는 PluginRegistry.count ≥ 1 및 훅 동작을 검증.

## 관련

- ADR-172 (PluginManifest + PluginLoader)
- SP-D.3 로드맵 G87 Plugin Registry Gate
