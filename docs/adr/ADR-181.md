# ADR-181: ZeroTrustAuditLog — HMAC-SHA256 Chain Audit Log

**Date**: 2026-05-28
**Status**: Accepted
**Phase**: SP-D.3 Zero-Trust Security (V720)

## Context

Every PASS/DENY decision by `ZeroTrustMiddleware` must be tamper-evidently logged. A simple append-only list is insufficient because records can be modified in memory or on disk without detection.

## Decision

Implement `ZeroTrustAuditLog` as an HMAC-SHA256 chained log:
- Each `AuditRecord` contains `chain_hash = HMAC-SHA256(prev_chain_hash + JSON(body), secret_key)`
- The first record uses `"GENESIS"` as `prev_hash`
- `verify_chain()` recomputes every hash and compares with `hmac.compare_digest` (timing-safe)
- When records are evicted due to `max_records`, the eviction anchor (`_eviction_anchor`) is updated to the last evicted record's hash, preserving chain verifiability for the retained window

## Key design choices

1. **Threading safety**: `_lock` mutex on all mutating and verifying operations
2. **Eviction anchor**: `_eviction_anchor` replaces GENESIS as the base for bounded logs
3. **Query interface**: filter by subject/tenant/decision/action with recency-first ordering
4. **`extra` field**: arbitrary metadata per record for extensibility

## Consequences

- Tamper detection: any in-memory modification of a record body or hash is caught by `verify_chain()`
- Integration: V721 Gate G88 will wire `ZeroTrustMiddleware.audit_log` → `ZeroTrustAuditLog.append()`
- Limitation: in-memory only; persistence layer (e.g., SQLite) is out-of-scope for SP-D.3
