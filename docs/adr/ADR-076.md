# ADR-076: RetrainingScheduler — F1 Drift 기반 재학습 스케줄러

## 상태
ACCEPTED

## 맥락
Phase C SP-C.1 에서 MetaLearner(V631)가 Bayesian Optimization으로 가중치를 탐색하고,
ConstitutionWeightTracker(V632)가 가중치 이력을 영속화하며,
PatternLibraryV2(V633)가 패턴을 압축/랭킹한다.
이 세 구성요소가 생성한 데이터는 궁극적으로 모델 재학습(LoRA finetune)의 근거가 되어야 한다.
그런데 "언제 재학습해야 하는가"에 대한 판단 기준이 없었다.
수동으로 재학습을 트리거하면 과도한 GPU 비용이 발생하거나 반대로 F1 저하를 방치하게 된다.

## 결정
**F1 drift 기반 자동 재학습 스케줄러 `RetrainingScheduler`를 도입한다.**

- **DRIFT_THRESHOLD = 0.03**: 현재 F1과 기준 F1의 절댓값 차이가 0.03 이상일 때 재학습 트리거
- **MIN_INTERVAL_DAYS = 7**: 마지막 재학습 이후 7일이 경과하지 않으면 스케줄 거부
- LOSDB JSONL 영속화: 스케줄 이력을 append-only 로 기록
- `should_retrain()` + `schedule()` 2단계 API

## 근거

### DRIFT_THRESHOLD = 0.03 선택
- Phase B 실험 데이터 기준, F1 ±0.03 이내는 측정 노이즈(sampling variance) 범위
- 0.03 미만 drift로 재학습 시 개선 없이 GPU 비용만 소모
- 0.05 이상으로 설정하면 실질적 성능 하락을 너무 오래 방치

### MIN_INTERVAL_DAYS = 7
- GPU SLO Phase C: 월 $200 하드 상한 (ADR 시스템 원칙)
- 7일 간격 = 최대 4~5회/월 재학습 → $200 범위 내 유지
- 긴급 상황은 `force=True` 옵션으로 우회 가능

### JSONL append-only 영속화
- V632 ConstitutionWeightTracker와 동일한 패턴으로 이력 보존
- 비파괴성 보장: 오래된 스케줄 이력 삭제 없이 감사 가능

## 결과

### 긍정적
- 재학습 시점 자동 판단으로 수동 개입 최소화
- GPU 비용 과소비 방지
- F1 drift 이력으로 모델 성능 추세 분석 가능

### 부정적/제약
- DRIFT_THRESHOLD 고정값이므로 작품 장르별 최적값이 다를 수 있음
- 실제 F1 측정 파이프라인(EquivalenceTester 연동)은 V635+ 에서 구현

## 대안 검토

| 방안 | 기각 이유 |
|------|-----------|
| 시간 기반 정기 스케줄 (매주 고정) | F1 변화와 무관 — 불필요한 재학습 발생 |
| F1 절대값 기준 (F1 < 0.75) | 기준선이 작품마다 달라 범용성 없음 |
| 수동 트리거 전용 | 자동화 목표에 반함 |
| drift + 추가 지표 조합 (entropy 등) | 복잡도 증가, V634 범위 초과 → Phase C 후속 버전 검토 |
