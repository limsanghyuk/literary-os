# ADR-067: SharedCharacterDB v2.0 + SharedWorldDB v2.0 설계 결정 (V607, SP-B.3 시작)

## 상태
**Accepted** — 2026-05-23

## 컨텍스트

SP-B.2(RLHF 루프, V601~V606) 완료 이후 SP-B.3 MultiWork 협업 레이어에 진입한다.
기존 `SharedCharacterDB` / `SharedWorldDB` (v1.0)는 기본적인 캐릭터·세계관 공유 기능을
제공하나, 다음 요구사항을 충족하지 못한다.

1. **RLHF 보상 추적**: PPOTrainer가 산출한 R(scene) 스코어를 캐릭터 단위로 기록·집계
2. **불변 스냅샷**: 동시 편집 시 롤백 가능한 체크포인트 이력 관리
3. **충돌 감지**: 두 프로젝트가 동일 캐릭터/로케이션을 다른 방향으로 수정할 때 자동 탐지
4. **일관성 지표**: 캐릭터 호 (arc) 와 세계관 타임라인의 내적 일관성을 정량화

## 결정

### D-1: 상속 기반 v2 확장 (비파괴적)

기존 v1 API를 일절 변경하지 않고 `SharedCharacterDBV2(SharedCharacterDB)` 형태로 상속.
- v1 사용자(기존 코드)는 수정 없이 동작 유지
- v2 전용 잠금(`_lock_v2: threading.Lock`)을 별도로 사용해 v1 내부 `_lock`과 데드락 방지
- `VERSION = "2.0.0"` 클래스 상수로 버전 식별

### D-2: 불변 스냅샷 (CharacterSnapshot / WorldSnapshot)

```python
@dataclass
class CharacterSnapshot:
    snapshot_id: str   # uuid4 앞 8자
    character_id: str
    label: str
    timestamp: float
    data: Dict[str, Any]   # 딥카피 저장

    def checksum(self) -> str:   # SHA-256(JSON) 앞 16자
```

- `checkpoint(character_id, label)` → snapshot_id 반환
- `restore(character_id, snapshot_id)` → 딥카피 복원, 이력 유지
- `list_snapshots(character_id)` → 시간순 정렬 목록

WorldSnapshot도 동일 패턴, 세계관 전체를 대상으로 함.

### D-3: RewardTrace — RLHF 보상 이력 통합

```python
@dataclass
class RewardTrace:
    character_id: str
    scores: List[float]
    timestamps: List[float]

    def mean(self) -> float: ...          # 전체 평균
    def trend(self) -> float: ...         # recent_5_mean − overall_mean
```

- `record_reward(character_id, reward)` → RewardTrace에 append
- `consistency_score(character_id)` 산식:
  - `arc_score = 1 / (1 + std(arc_values))` — arc_values 분산이 낮을수록 일관
  - `reward_score = (mean_reward + 10) / 20` — [−10, +10] → [0, 1] 정규화
  - `final = (arc_score + reward_score) / 2`

### D-4: 충돌 감지 프로토콜 (ConflictRecord / LocationConflict)

두 프로젝트 A, B가 동일 개체를 수정했을 때 충돌로 판정:
```
changed_a = hash(current_A) != hash(base_A)
changed_b = hash(current_B) != hash(base_B)
bases_differ = base_A_hash != base_B_hash
conflict = changed_a AND changed_b AND bases_differ
```

- `register_project_state(project_id, entity_id)` → 기준 해시 등록
- `detect_conflicts(entity_id, project_a, project_b)` → ConflictRecord or None
- `resolve_conflict(conflict_id)` → resolved=True 마킹

### D-5: WorldDB 일관성 지표 — 타임라인 간격 균일도

```
gaps = [e2.timestamp − e1.timestamp, ...]  # 연속 이벤트 간 간격
consistency_score = 1 / (1 + std(gaps) / mean(gaps))
```

이벤트가 0개 또는 1개면 1.0 반환 (완전 일관).

### D-6: JSON 직렬화 (export / import)

모든 v2 내부 상태를 `export_snapshot() → Dict` / `import_snapshot(data) → int` 로
직렬화·역직렬화. 세션 간 지속성 및 단위 테스트 픽스처 생성을 지원.

## 결과

- **Gate 영향**: G58(V613 예정)은 본 ADR 이후 v2 API를 기반으로 평가
- **테스트**: `tests/unit/test_v607_multiwork_v2.py` 25+ TC로 모든 public 메서드 커버
- **비파괴**: v1 테스트 전량 유지, 신규 TC만 추가
- **버전**: v10.12.0 (ADR-067 + V607)

## 대안 고려

| 대안 | 거부 이유 |
|------|-----------|
| v1 직접 수정 | 기존 테스트 전량 재작성 필요, 리스크 과다 |
| 별도 독립 클래스 | 공유 잠금 로직 중복, 유지보수 비용 증가 |
| SQLite 영속화 | 이 단계에선 과잉설계; Phase C에서 재검토 |
| KL 기반 일관성 | 실시간 계산 비용 과다; std 기반으로 충분 |
