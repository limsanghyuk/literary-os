# ADR-035: LLM Adapter Canonical — G3(adapters_live) 단일 표준화

| 항목 | 내용 |
|------|------|
| **상태** | Accepted |
| **버전** | V577 |
| **날짜** | 2026-05-19 |
| **결정자** | 아키텍처 위원회 (CSA · CSC · CSPE) |
| **관련 ADR** | ADR-031 (LLM0StaticGate), ADR-034 (V575 보안패치) |

---

## 배경

literary-os 코드베이스에는 4세대의 LLM 어댑터가 공존하고 있다.

| 세대 | 위치 | 인터페이스 | 상태 |
|------|------|-----------|------|
| G1_root | `llm_bridge/*.py` | `LLMBridgeInterface` | 구세대 |
| G1_sub  | `llm_bridge/adapters/` | `LLMBridgeInterface` | 구세대 |
| G2      | `llm_bridge/adapters_v2.py` | `AdapterContractV2` | 구세대 |
| G3      | `adapters_live/` | `.call(ctx)` | **캐노니컬** |

G3 어댑터(`RealClaudeAdapter`, `RealOpenAIAdapter`, `RealOllamaAdapter`)는 다음 이유로 캐노니컬로 선정되었다.

1. **멀티테넌트 지원**: `tenant_id` 파라미터로 테넌트별 격리 보장
2. **LLM-0 원칙 준수**: `call_fn` 주입으로 CI 환경에서 실 API 호출 차단
3. **지수 백오프**: 재시도 로직 내장 (max_retries, base_delay, max_delay)
4. **비용 추적**: `RealLLMResponse.cost_usd` 필드로 호출당 비용 기록
5. **tiktoken 통합**: 정확한 토큰 카운팅 (미설치 시 len//4 폴백)

## 문제

G3 어댑터의 인터페이스는 `.call(ctx) -> RealLLMResponse`이며, 시스템 전반에서
사용되는 `LLMBridgeInterface.generate(prompt, ctx) -> str`와 불일치한다.

이로 인해:
- `UnifiedLLMGateway.make_default_gateway()`가 여전히 G1 어댑터를 직접 사용
- G2/G3 어댑터가 병렬로 존재하여 어느 것이 표준인지 불명확
- 신규 기능 추가 시 어느 세대를 기준으로 구현해야 하는지 혼란 발생

## 결정

### 1. G3 캐노니컬 선정

`adapters_live/` 디렉터리의 어댑터를 공식 캐노니컬로 지정한다.

### 2. CanonicalLLMBridge 래퍼 신설

`literary_system/llm_bridge/canonical_adapter.py`에 `CanonicalLLMBridge` 클래스를 구현한다.

- **역할**: G3 `.call(ctx)` → `LLMBridgeInterface.generate(prompt, ctx)` 브릿지
- **구현 방식**: Adapter Pattern — `generate()` 내부에서 `ctx.extra["user_prompt"] = prompt`
  주입 후 `g3_adapter.call(ctx)` 호출
- **팩토리 함수**: `make_canonical_claude()`, `make_canonical_ollama()`, `make_canonical_openai()`

```python
# 사용 예시
bridge = make_canonical_claude(model="claude-haiku-4-5-20251001", call_fn=mock_fn)
text = bridge.generate("이야기를 생성하라", ctx)
```

### 3. UnifiedLLMGateway 갱신

`make_default_gateway()`가 G1 어댑터 대신 `CanonicalLLMBridge`를 반환하도록 갱신한다.

```python
# 변경 전 (G1)
from literary_system.llm_bridge.claude_adapter import ClaudeAdapter
providers["quality"] = ClaudeAdapter("claude-sonnet-4-6", ...)

# 변경 후 (G3 canonical)
from literary_system.llm_bridge.canonical_adapter import make_canonical_claude
providers["quality"] = make_canonical_claude(model="claude-sonnet-4-6", call_fn=call_fn)
```

### 4. G1/G2 Deprecation 경고

G1/G2 어댑터 `__init__` 메서드에 `logging.warning()` 추가:

```python
import logging
logging.warning(
    "[DEPRECATED V577] %s는 G1/G2 구세대 어댑터입니다. "
    "V578 이후 제거 예정. CanonicalLLMBridge 사용을 권장합니다.",
    self.__class__.__name__,
)
```

### 5. Gate G35 AdapterCanonical 신설

`release_gate.py`에 G35 게이트를 추가하여 다음을 검증:
- `CanonicalLLMBridge`가 `LLMBridgeInterface` 구현체임을 확인
- `make_canonical_claude(call_fn=mock_fn).generate()` 정상 동작 확인
- `UnifiedLLMGateway.make_default_gateway()`가 `CanonicalLLMBridge` 반환 확인

## 결과

### 긍정적 결과
- 어댑터 세대 혼란 해소: G3 단일 표준
- `LLMBridgeInterface` 계약 일관성 유지
- G3의 멀티테넌트/비용추적/LLM-0 특성 전파

### 부정적 결과
- G1/G2를 직접 사용하는 소수 모듈(drama_episode_generator, gate17, gate21)의
  점진적 마이그레이션 필요 (V578~V579 예정)
- `CanonicalLLMBridge`가 한 겹의 간접 계층을 추가하나, 성능 영향 무시 가능

## 준수 확인

| 원칙 | 준수 여부 |
|------|----------|
| LLM-0 (외부 LLM 호출 금지) | ✅ call_fn 주입으로 CI 보호 |
| DEV_MODE 기본값=false (ADR-034) | ✅ 어댑터 직접 연관 없음 |
| Gate 연속성 | ✅ G35 AdapterCanonical 추가 |
| 하위 호환 | ✅ G1/G2 즉시 제거 없음, Deprecation 경고만 |
