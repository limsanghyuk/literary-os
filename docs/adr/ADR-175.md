# ADR-175: PluginLifecycleManager — 플러그인 생명주기 상태 기계

**날짜**: 2026-05-28  
**상태**: Accepted  
**버전**: V714  
**작성자**: Literary OS Engineering

---

## Context

SP-D.3 Plugin 서브시스템(V711~V716)에서 플러그인은 등록(PluginRegistry) 이후 동적으로
활성화·비활성화될 수 있어야 한다. 단순 boolean 플래그가 아닌 정형 상태 기계가 필요하며,
각 전환 시 훅 콜백을 통해 외부 시스템(모니터링, 감사 로그)이 이벤트를 수신해야 한다.

---

## Decision

`PluginLifecycleManager` 클래스를 `literary_system/plugins/plugin_lifecycle.py`에 구현한다.

### 상태 기계

```
INACTIVE ──activate()──► ACTIVATING ──성공──► ACTIVE
                                     └─실패──► INACTIVE_ERROR

ACTIVE ──deactivate()──► DEACTIVATING ──► INACTIVE

ACTIVE ──restart()──► (deactivate 순서대로) ──► RESTARTING ──activate()──► ACTIVE
```

### 주요 설계 결정

1. **상태 열거체**: `LifecycleState(str, Enum)` — JSON 직렬화 호환 문자열 기반
2. **레코드 분리**: 상태·오류·활성화 횟수를 `LifecycleRecord` 데이터클래스로 캡슐화
3. **엔트리포인트 검증**: `activate()` 시 `importlib.import_module(top_module)` 실제 임포트로 검증
   - 이유: PluginWhitelist는 샌드박스 전용이며 literary_system 패키지 자체를 화이트리스트에 포함하지 않아
     `validate_import()` 사용 시 자기 모듈이 INACTIVE_ERROR로 분류되는 문제가 있었음
4. **restart() 훅 순서**: `deactivate()` 먼저 실행 후 RESTARTING 상태 전환
   - 이유: `rec.state = RESTARTING` 먼저 설정 시 `deactivate()` 내 `if rec.state != ACTIVE` 가드에
     막혀 on_deactivate 훅이 실행되지 않음
5. **훅 격리**: 훅 실패가 생명주기 전환을 막지 않도록 try/except로 격리
6. **독립 네임스페이스**: 각 레코드는 독립적이며 registry/sandbox 공유만 허용

### API

```python
manager = PluginLifecycleManager(registry=reg, sandbox=sb)
manager.on_activate(lambda pid, manifest: ...)
manager.on_deactivate(lambda pid: ...)
manager.on_error(lambda pid, exc: ...)

rec = manager.activate("my-plugin")   # LifecycleRecord
manager.deactivate("my-plugin")
manager.restart("my-plugin")

manager.list_active()    # List[str]
manager.list_errored()   # List[str]
manager.active_count     # int
```

---

## Consequences

- ✅ 플러그인 상태를 정형화하여 모니터링·감사 로그가 이벤트 기반으로 수신 가능
- ✅ 오류 플러그인이 다른 플러그인의 활성화를 차단하지 않음 (독립 레코드)
- ✅ restart()가 on_deactivate → on_activate 전체 훅 체인을 올바르게 실행
- ⚠️ 상태 전환 원자성은 보장하지 않음 (단일 스레드 가정; 멀티스레드 지원은 V7xx에서 추가 예정)
