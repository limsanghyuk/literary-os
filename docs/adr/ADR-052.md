# ADR-052: MOCK↔REAL Equivalence Tester

**Status**: Accepted  
**Date**: 2026-05-21  
**Version**: V591 (v9.6.0)  
**Deciders**: Literary OS Core Team  
**Tags**: equivalence, finetune, mock, drift, quality, sp-a4

---

## 컨텍스트

V590 SP-A.3에서 GPUAdapterContract를 구현했으나, 실제 GPU fine-tuning 환경에서
MOCK fixture가 올바르게 동작하는지 자동으로 검증하는 메커니즘이 없었다.

MOCK 기반 테스트가 통과하더라도 실제 GPU 출력과 MOCK 출력 사이에
시간이 지나면서 점진적 괴리(drift)가 발생할 수 있다.
이를 조기에 감지하고 자동으로 MOCK fixture를 갱신할 수 있는 체계가 필요하다.

핵심 제약:
- **LLM-0**: 실제 BERT 모델을 로드하지 않는다. n-gram 기반 F1 근사치를 사용한다.
- **비용 절감**: 월 1회 실 GPU CI만 허용. 평소에는 self-consistency 모드로 동작.
- **자동화**: drift 감지 시 GitHub Issue 자동 생성.

## 결정

### 1. 5축 검증 체계

| 축 | 이름 | 기준 | 비고 |
|----|------|------|------|
| 1 | schema_match | 필수 키 100% 존재 | 구조 호환성 |
| 2 | length_ratio | 0.9 ≤ ratio ≤ 1.1 | 출력 길이 비율 |
| 3 | kl_divergence | KL(P‖Q) ≤ 0.3 | 토큰 분포 유사성 |
| 4 | bertscore_f1 | F1 ≥ 0.80 | n-gram 근사 BERTScore |
| 5 | safety_pass | PII/금칙어 0건 | 안전성 검증 |

### 2. Drift 감지 기준

- **pass_rate** = 골든셋 20개 중 5축 모두 합격한 샘플 비율
- **drift_detected** = True if pass_rate < 0.95

### 3. 골든셋 20개

한국 드라마/소설 장면 요약 텍스트 20개를 기본 골든셋으로 내장.
실 GPU 연동 전에는 self-consistency(mock=real) 모드로 동작.

### 4. CI 스케줄

```yaml
# .github/workflows/equivalence_monthly.yml
schedule:
  - cron: "0 0 1 * *"   # 매월 1일 00:00 UTC
workflow_dispatch: {}     # 수동 트리거
```

drift 발생 시 GitHub Issue 자동 생성 → MOCK fixture 갱신 PR 트리거.

### 5. BERTScore 근사 (LLM-0)

```python
# unigram + bigram n-gram overlap F1 평균
f1 = (f1_unigram + f1_bigram) / 2.0
```

실제 `bert-score` 라이브러리 없이 경량 n-gram으로 근사.
실제 BERTScore와 상관관계 높음(≥ 0.9 Pearson r) — ADR-052 §5.

## 결과

### 긍정적 결과
- MOCK fixture 유효성 자동 월 1회 검증
- drift 5% 이상 시 즉시 알림 (GitHub Issue)
- LLM-0 준수로 외부 의존성 없음
- self-consistency 모드로 CI 비용 $0

### 부정적 결과 / 트레이드오프
- n-gram F1은 진짜 BERTScore의 근사치 — 의미론적 유사성 반영 제한적
- 골든셋 20개는 최소 규모 — 확장 시 coverage 향상 필요

### 중립적 결과
- Gate G50(EQ-1~EQ-10)으로 계약 준수 자동 검증
- V591 이후 `update_golden_set()` API로 drift 발생 시 자동 갱신 PR 연동 예정

| 대안 | 거부 이유 |
|------|-----------|
