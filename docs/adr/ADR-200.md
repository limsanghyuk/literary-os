# ADR-200: Phase E 인프라 전략 — Helm / KEDA / ArgoCD

| 항목 | 내용 |
|------|------|
| 상태 | Accepted |
| 결정자 | CSA · CSC · CSPE |
| 날짜 | 2026-05-29 |
| 관련 | ADR-201, D-M-12, V738~V740 |

## 맥락

Literary OS v13.0.0 이후(Phase E)에서 클라우드-네이티브 배포를 지원하기 위해
컨테이너 오케스트레이션·자동 스케일링·GitOps 3종 기술 스택을 사전 정의한다.

Phase D 설계도(D-M-12)는 "Phase E KEDA/ArgoCD manifest를 V740에서 스텁으로 포함"할 것을
명시하며, 이는 Phase E 진입 시 인프라 설계 결정을 재검토 없이 진행하기 위함이다.

## 결정

### 1. 패키지 매니저: Helm 3

- Literary OS의 Kubernetes 배포 단위는 단일 Helm Chart `literary-os`(version 13.0.0)
- `values.yaml`에서 FL(Federated Learning) 설정(`fl.enabled`, `fl.minClients`, `fl.dpEpsilon`)을 포함
- 환경별 오버라이드는 `values-{env}.yaml` 패턴 사용 (Phase E에서 구현)

### 2. 오토스케일링: KEDA 2.x

- CPU/Memory 기본 스케일링 외 Prometheus 메트릭(`literary_os_request_queue_depth`)을 기반으로 스케일
- `minReplicaCount: 2` / `maxReplicaCount: 20` — 고가용성과 비용 상한을 동시에 보장
- TriggerAuthentication은 `literary-os-keda-sa` ServiceAccount 기반

### 3. GitOps: ArgoCD

- 단일 `Application` 리소스가 `deploy/phase_e/helm/literary-os`를 추적
- `automated.selfHeal: true` + `prune: true` → git 상태와 클러스터 상태 자동 동기화
- `AppProject`로 소스 레포지토리 및 배포 네임스페이스(`literary-os`, `literary-os-staging`) 제한

## 결과

- Phase E 진입 시 인프라 결정 재논의 불필요 → 개발 속도 향상
- Helm + KEDA + ArgoCD 조합은 커뮤니티 표준으로 장기 유지보수성 확보
- 스텁 단계이므로 실 클러스터 연결 없이 CI 검증 가능 (PhaseEManifestValidator)

## 기각된 대안

| 대안 | 이유 |
|------|------|
| Kustomize | Helm에 비해 values 관리 복잡, 팀 친숙도 낮음 |
| HPA only | Prometheus 기반 커스텀 메트릭 스케일링 불가 |
| Flux | ArgoCD 대비 UI 부재, 현 팀 운영 경험 부족 |
| gRPC manifest 포함 | ADR-199 결정: gRPC는 Phase E에서 별도 ADR |
