# ADR-204: RPO/RTO 정책 — Literary OS 재해복구 기준 (V743)

| 항목 | 내용 |
|------|------|
| 상태 | Accepted |
| 결정자 | CSA · CSC · CSPE |
| 날짜 | 2026-05-29 |
| 관련 | ADR-202, ADR-203, ADR-192 §RS-09 |

## 맥락

엔터프라이즈 SLA에서 요구하는 재해복구 지표 두 가지:

- **RPO (Recovery Point Objective)**: 마지막 백업에서 장애까지의 최대 허용 데이터 손실 시간
- **RTO (Recovery Time Objective)**: 장애 발생에서 서비스 복원까지의 최대 허용 시간

## 결정

### RPO: 1시간 (3600초) — 하드 제한

```
DRBackupManager(backup_interval_seconds ≤ 3600) 강제
```

- 모든 테넌트에 동일하게 적용
- 코드 레벨 강제 (ValueError): 설정 오류를 배포 전에 차단

### RTO: 1시간 (3600초) — 소프트 목표 (Phase D)

```
DRRestoreManager(max_restore_seconds=3600)  # 소프트 목표
```

- Phase D에서는 `duration_seconds` 로깅만 (하드 강제 X)
- Phase E에서 실제 스토리지 연결 시 하드 강제로 업그레이드 예정

### 보존 정책

- 기본 24개 백업 보존 (1h 간격 × 24 = 24h 이력)
- 프루닝: 가장 오래된 항목부터 제거

### 테넌트별 독립 RPO

- 테넌트별 별도 `rpo_compliant()` 확인
- 하나의 테넌트 RPO 실패가 다른 테넌트에 영향 없음

## 결과

- G91 Gate에서 RPO 정책 준수 자동 검증
- Phase E 진입 시 RTO 하드 강제화 ADR 업데이트 예정

## 기각된 대안

| 대안 | 이유 |
|------|------|
| RPO 30분 | 백업 빈도 2배 증가, 시스템 부하 |
| RPO 4시간 | 엔터프라이즈 SLA C-03 기준 미달 |
| RTO 하드 강제 (Phase D) | 실 스토리지 없어 실측 불가 |
