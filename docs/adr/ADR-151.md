# ADR-151: W3C TraceContext Level 1 Propagator 도입 (D-M-02)

## Status
Accepted — V688 (2026-05-28)

## Context
SP-D.1 관측가능성 기반 구축의 핵심 요건으로, 마이크로서비스 간 트레이스 컨텍스트 전파 표준이 필요하다.

### 문제
- literary-os의 요청 경계(HTTP 헤더, 메시지 큐, RPC 등)를 넘나드는 분산 트레이싱이 불가능
- 독자적 trace ID 포맷은 외부 APM 도구(Jaeger, Zipkin, Tempo)와 호환 불가
- Span 부모-자식 관계를 전파할 공식 메커니즘 부재

### 요건 (D-M-02)
| 요건 | 상세 |
|------|------|
| D-M-02-a | W3C Trace Context Level 1 traceparent 헤더 파싱/생성 |
| D-M-02-b | tracestate 헤더 보존 |
| D-M-02-c | TraceContext inject / extract API |
| D-M-02-d | child_context()로 span 계층 생성 |
| D-M-02-e | OTel SDK Adapter (in-memory mock) 제공 |

## Decision

### 1. W3C Trace Context Level 1 채택
`traceparent` 헤더 포맷: `{version}-{trace_id}-{parent_id}-{flags}`

```
version    : "00" (고정, 2-char hex)
trace_id   : 32-char lowercase hex (128-bit, secrets.token_hex(16))
parent_id  : 16-char lowercase hex (64-bit, secrets.token_hex(8))
flags      : 2-char hex (TraceFlags IntFlag: SAMPLED=0x01)
```

### 2. 구현 모듈
```
literary_system/ops/trace_context.py   ← TraceContext dataclass + Propagator
literary_system/ops/otel_adapter.py    ← OtelSdkAdapter + SpanExporter (in-memory)
```

### 3. 핵심 API
```python
# 새 루트 컨텍스트 생성
ctx = new_trace_context(sampled=True)

# 자식 span 컨텍스트 생성 (trace_id 상속, 새 span_id)
child = child_context(parent_ctx)

# HTTP 헤더 inject
TraceContextPropagator.inject(ctx, headers)   # → headers["traceparent"] = "00-..."

# HTTP 헤더 extract
ctx = TraceContextPropagator.extract(headers)  # None if missing/invalid

# OTel Adapter
adapter = create_otel_adapter(service_name="literary-os")
with adapter.trace("generate_chapter") as span:
    span.set_attribute("chapter", 3)
    # ... 작업 수행
```

### 4. TraceFlags 설계
```python
class TraceFlags(IntFlag):
    NONE    = 0x00
    SAMPLED = 0x01
```
- `IntFlag` 사용: 비트 연산 자연스럽게 지원
- Phase D에서 향후 DEFERRED(0x02), DEBUG(0x04) 플래그 추가 가능

### 5. SpanExporter (in-memory)
```python
class SpanExporter:
    def export(self, span: SpanData) -> None  # 내부 리스트에 축적
    def spans: List[SpanData]                 # 조회
    def clear_spans(self) -> None             # 초기화
```
- Production에서는 OTLP gRPC exporter로 교체 (ADR-152 이후)

## Consequences

### 긍정적
- Jaeger/Zipkin/Grafana Tempo와 완전 호환
- 표준 헤더로 API Gateway, Load Balancer 로그와 자동 상관
- in-memory exporter로 테스트 100% 오프라인 가능

### 부정적/트레이드오프
- 외부 OTel SDK 라이브러리 불도입 (in-memory mock): Production 배포 시 교체 필요
- tracestate는 투명 보존만, 내부 파싱 없음
