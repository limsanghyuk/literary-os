# ADR-180: ZeroTrustMiddleware — Request Interceptor Design

**Date**: 2026-05-28
**Status**: Accepted
**Deciders**: Literary OS Platform Team
**Phase**: SP-D.3 Zero-Trust Security (V719)

## Context

G88 Zero-Trust Security requires every internal service call to be authenticated and authorized before processing. Modules issuing requests (e.g., plugin executors, API routers) must not implement auth logic individually — a single interceptor must enforce the full policy uniformly.

## Decision

Implement `ZeroTrustMiddleware` as a composable request interceptor combining `ZeroTrustTokenService` (V717) and `TenantAuthority` (V718) into a single `process(request)` entry point.

### Key design choices

1. **`ZTRequest` dataclass** — bearer token + tenant context + optional body; decoupled from HTTP framework.
2. **Fail-closed** — missing token → 401; expired → 401; validation error → 401; unauthorized → 403. Only explicit PASS → 200.
3. **Audit log** — every decision (PASS/DENY) appended to `self.audit_log` list with timestamp, subject, reason.
4. **Event hooks** — `on_pass(fn)` / `on_deny(fn)` for downstream telemetry without subclassing.
5. **Counters** — `pass_count()` / `deny_count()` for dashboard metrics.

## Consequences

- **Positive**: Central enforcement, testable in isolation, zero framework coupling.
- **Negative**: Caller must adapt framework-native request objects to `ZTRequest` (thin adapter needed per framework).
- **Neutral**: `security/` package currently isolated; connection expected in V721 Gate wiring.
