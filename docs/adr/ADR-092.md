# ADR-092: V625 biweekly_train + Lambda 폴백 + 자동 복구 전략

**날짜**: 2026-05-25  
**상태**: Accepted  
**작성자**: Literary OS AI 개발팀  
**버전**: v10.30.0 (V625)

---

## 배경

본안 v3.0 V625 계획에 따라 LoRA+RLHF 격주 자동 훈련 파이프라인(biweekly_train)과 RunPod 
가용성 부족 시 Lambda Labs H100으로 자동 전환하는 폴백 메커니즘을 도입한다.

R5d-1 보강 사항: biweekly_train CI + Lambda 폴백을 통해 GPU 공급 단절 없이 훈련 사이클을 보장한다.

---

## 결정

### 1. biweekly_train.yml CI 워크플로우

- **트리거**: 매월 1·15일 02:00 KST (`cron: '0 17 1,15 * *'`) + `workflow_dispatch`
- **기본값**: `dry_run=true` (실제 GPU 기동은 별도 승인 필요)
- **3-Job 구조**:
  1. `check-runpod` — RunPod 가용성 확인
  2. `train` — RunPod 또는 Lambda 폴백 실행
  3. `post-train-gate` — 훈련 후 Gate 회귀 검증

### 2. check_runpod_availability.py

- `RunPodChecker` 클래스: RunPod API 가용성 조회
- 폴백 GPU 타입 순차 확인: RTX_4090 → A100_SXM → RTX_3090 → A6000
- dry_run 모드 지원 (CI 테스트용)

### 3. notify_slack.py

- `SlackNotifier` 클래스: Incoming Webhook 알림
- 3개 level: `info` / `warn` / `error`
- 채널: `#losdb-ops` (기본)

### 4. AutoRecoveryScheduler (literary_system/finetune/lora_job_runner.py 확장)

- 백엔드 자동 선택 (`decide_backend(runpod_available)`)
- 시도 기록 + 에스컬레이션 판단 (`should_escalate()`)
- 최대 재시도: 3회 (초과 시 Slack 에스컬레이션)
