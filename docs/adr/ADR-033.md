## 맥락

V572 독립 감사(literary_os_v572_audit_proposal.docx)에서 다음 패턴의 버그가 발견되었다:

```python
# release_gate.py (버그 상태)
from literary_system.graph_intelligence.asd.narrative_debt_detector import DebtReport
#                                                                              ^^^^^^^^
#  실제 클래스명: NarrativeDebtReport — 이 import는 ImportError를 일으킴
```

`DebtReport`는 존재하지 않는 클래스명이다. 결과적으로 Gate28이 except 분기만 타게 되어
실질적으로 무음 비활성화 상태였다.

Preflight Step13은 `tests/` 디렉터리의 외부 패키지 의존성을 점검하지만,
`release_gate.py` Gate 함수 내 내부 타입명 오탐은 검사하지 못했다.

---

## 결정

`tools/preflight_step14.py`를 신설하여 다음을 자동 대조한다:

1. `release_gate.py`의 모든 `_gate_*` 함수 내부에서 alias 없이 사용되는
   `from literary_system.X import Y` 수집
2. 대상 모듈 `literary_system/X.py`의 최상위 class 및 function 이름 목록 추출
3. `Y`가 실제 이름 목록에 없으면 `MISMATCH` 보고

CI 파이프라인에 `preflight-step14` 잡을 추가하여:
- preflight(Step13) 성공 후 실행
- Step14 실패 시 test 잡 전체 블로킹 (`needs: [preflight, preflight-step14]`)

---

## 설계 결정 사항

### alias 있는 import 제외

```python
from literary_system.gates.gate9_drse_quality import _gate_drse_quality as _run
```

`as _run`이 있으면 `_gate_drse_quality`는 직접 사용되지 않으므로 검사 제외.
오직 alias 없는 직접 사용 import만 대조한다.

### class + function 모두 검사

내부 모듈에서 class 외에 top-level function도 import할 수 있으므로,
대상 모듈의 최상위 `ClassDef` + `FunctionDef` 모두 수집한다.

## 대안 검토

| 대안 | 거부 이유 |
|------|-----------|
| Step13 확장 | 책임 분리 원칙 위반 (Step13 = 외부 패키지, Step14 = 내부 타입명) |
| 런타임 import 시 자동 검출 | 이미 try/except로 무음 처리되어 CI에서 표면화 불가 |
| mypy 타입 검사 | 동적 import 패턴에 불안정 + 기존 빌드 체계에 mypy 없음 |

## 결과

- 동일 유형 타입명 오염 버그가 PR 단계에서 자동 탐지 가능
- `--strict` 모드: CI 블로킹
- 현재 검사 범위: 29개 Gate 함수, 55개 타입 import (V573 기준)
- V573 수정 후 불일치 0건 확인
