# ADR-176: Plugin SDK — BasePlugin + PluginContext + PluginTestHarness

**날짜**: 2026-05-28  
**상태**: Accepted  
**버전**: V715  
**작성자**: Literary OS Engineering

---

## Context

SP-D.3 Plugin 서브시스템(V711~V716)에서 외부 개발자가 플러그인을 작성할 수 있어야 한다.
V711~V714에서 내부 인프라(Manifest, Registry, Sandbox, Lifecycle)를 구축했으므로,
이제 플러그인 개발자가 실제로 사용하는 퍼블릭 API 계층이 필요하다.

---

## Decision

세 개의 SDK 컴포넌트를 `literary_system/plugins/`에 추가한다.

### 1. PluginContext

런타임에 플러그인에 주입되는 컨텍스트 객체.

```python
ctx = PluginContext(
    plugin_id="my-plugin",
    permissions=frozenset([PluginPermission.READ_CORPUS]),
    metadata={"project": "drama-A"},
)
ctx.emit("output text")
ctx.require_permission(PluginPermission.READ_CORPUS)
```

**설계 원칙**:
- emit()은 str만 허용 (TypeError 발생)
- get_outputs()는 내부 리스트의 복사본 반환 (불변성 보장)
- require_permission()은 권한 없을 시 PermissionError 발생

### 2. BasePlugin

플러그인 개발자가 상속하는 추상 기반 클래스.

```python
class MyPlugin(BasePlugin):
    MANIFEST = PluginManifest(
        plugin_id="my-plugin",
        name="My Plugin",
        version="1.0.0",
        entry_point="my_package.my_plugin.MyPlugin",
    )

    def on_activate(self) -> None:
        self.context.emit("hello!")

    def on_deactivate(self) -> None:
        pass
```

**설계 원칙**:
- `__init_subclass__` 훅으로 MANIFEST 선언을 클래스 생성 시 즉시 검증
- 추상 메서드: `on_activate()`, `on_deactivate()` (필수 구현)
- 선택적 오버라이드: `on_error(exc)` (기본 구현 = 무시)

### 3. PluginTestHarness

테스트 코드에서 플러그인을 격리 실행하는 헬퍼.

```python
harness = PluginTestHarness(
    MyPlugin,
    permissions=frozenset([PluginPermission.READ_CORPUS]),
)
plugin = harness.activate()
assert harness.outputs == ["hello!"]
harness.deactivate()
```

**설계 원칙**:
- 실제 Registry/LifecycleManager 없이 플러그인 로직만 테스트 가능
- on_activate 예외 → last_error에 저장, is_activated=False
- run_in_sandbox()로 코드 격리 실행도 지원

---

## Consequences

- ✅ 외부 플러그인 개발자가 내부 인프라를 몰라도 플러그인 작성 가능
- ✅ PluginTestHarness로 플러그인 단위 테스트 작성 용이
- ✅ MANIFEST 누락을 런타임이 아닌 클래스 정의 시 즉시 감지
- ⚠️ PluginContext는 단일 스레드 가정 (thread-safe 아님)
