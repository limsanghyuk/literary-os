# ADR-174: PluginWhitelist + PluginSandbox — 모듈 화이트리스트 및 코드 실행 격리

**상태**: Accepted  
**날짜**: 2026-05-28  
**버전**: V713  
**결정자**: SP-D.3 보안 설계팀

---

## 맥락

외부 플러그인이 임의 Python 코드를 실행할 수 있으면 `os`, `subprocess`, `socket` 등을 통해
호스트 시스템에 접근하거나 데이터를 외부로 유출할 수 있다.
PluginLoader(V711)는 `importlib`로 플러그인을 로드하지만, 실행 시점 코드 격리가 없다.
V713에서 두 계층의 방어를 추가한다.

## 결정

### PluginWhitelist

모듈 허용 목록을 명시적으로 관리한다.

- `DEFAULT_ALLOWED_MODULES`: 표준 라이브러리 안전 부분집합 (math, json, re, datetime 등)
- `BLOCKED_MODULES`: 변경 불가 차단 목록 (os, sys, subprocess, socket, ctypes 등)
- `is_allowed(module_name)` → BLOCKED 우선, 부모 패키지 매칭 지원

### PluginSandbox

RestrictedPython 기반 코드 실행 격리.

- `compile_restricted()`: AST 수준에서 위험 구문 컴파일 차단
- `__import__` 후킹: PluginWhitelist.is_allowed() 기반 동적 차단
- `PrintCollector`: 표준 출력 캡처 (64 KB 제한)
- `SandboxResult` 반환: success, return_value, stdout, error, locals_snapshot
- 각 `execute()` 호출은 독립 네임스페이스 → 상태 누출 없음

## 대안

- **AST 트리 직접 파싱**: 구현 복잡도가 높고 RestrictedPython이 이미 처리.
- **컨테이너 격리 (Docker)**: V713 범위를 초과, 향후 고려.

## 결과

- G87 Gate에서 PluginSandbox.execute() 격리 검증 포함.
- V714 이후 PluginLifecycle은 PluginSandbox를 통해 플러그인 코드 실행.
