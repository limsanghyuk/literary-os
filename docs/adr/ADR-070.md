# ADR-070: MultiWorkCIM v2.0 — 팩토리 패턴 + 업그레이드 유틸리티

- **상태**: Accepted
- **날짜**: 2026-05-23
- **구현**: V610
- **파일**: `literary_system/multiwork/multi_work_cim.py`

---

## 맥락

V609에서 `MultiWorkCIMV2`(v2 클래스)가 구현되어 `multi_work_cim_v2.py`에 분리됨.
기존 `multi_work_cim.py`의 `MultiWorkCIM`(v1)은 여전히 레거시 API 진입점으로 사용되고 있어,
v1/v2 사용 진입점이 두 파일로 분산됨.

V610에서는 `multi_work_cim.py`를 **v2.0으로 격상**하여 단일 공개 API 진입점을 정립한다.

---

## 결정

### D-1: CIMVersion 열거형 도입

```python
class CIMVersion(str, Enum):
    V1 = "v1"
    V2 = "v2"

    @classmethod
    def current(cls) -> CIMVersion:
        return cls.V2
```

`CIMVersion.current()` → 항상 V2 반환. 새 코드는 v2를 기본으로 사용.

### D-2: `ProjectCIM.to_v2()` — 비파괴적 v1→v2 마이그레이션

- v1 `entries` dict를 `CIMEntryV2`로 변환하여 새 `ProjectCIMV2` 인스턴스에 주입
- `reward_weighted_weight = weight × 0.5` (마이그레이션 기본값, 보상 정보 미확인 상태)
- 원본 v1 인스턴스 변경 없음 (비파괴)

### D-3: `MultiWorkCIM.upgrade_to_v2()` — 전체 인스턴스 변환

- `_project_cims` 전체를 순회하여 각각 `ProjectCIM.to_v2()` 호출
- 결과 `ProjectCIMV2` 인스턴스를 `MultiWorkCIMV2._project_cims_v2`와 `_project_cims`에 동시 주입
- 원본 `MultiWorkCIM` 인스턴스는 유지 (새 `MultiWorkCIMV2` 반환)

### D-4: `create_multi_work_cim()` 팩토리

```python
def create_multi_work_cim(
    version: str = "v2",   # 기본값 v2
    decay: float = 0.95,
    char_db = None,
    conflict_threshold: float = 0.30,
) -> MultiWorkCIM | MultiWorkCIMV2:
    ...
```

- `version='v2'`(기본): `MultiWorkCIMV2` 반환
- `version='v1'`: `MultiWorkCIM` 반환
- v2 모듈은 **지연 임포트(lazy import)** — circular import 방지

### D-5: Circular Import 방지

`multi_work_cim_v2.py`는 `multi_work_cim.py`를 임포트하므로,
역방향 임포트는 `TYPE_CHECKING` 블록 + 함수 내 지연 임포트로 처리:

```python
if TYPE_CHECKING:
    from .multi_work_cim_v2 import MultiWorkCIMV2, ProjectCIMV2

def upgrade_to_v2(self):
    from .multi_work_cim_v2 import MultiWorkCIMV2  # lazy
    ...
```

### D-6: `get_cim_version()` 헬퍼

```python
def get_cim_version(cim: MultiWorkCIM) -> CIMVersion:
    if hasattr(cim, "VERSION") and cim.VERSION.startswith("2"):
        return CIMVersion.V2
    return CIMVersion.V1
```

런타임에 임의의 CIM 인스턴스가 v1인지 v2인지 판별.

## 하위 호환성 보장

| 항목 | 상태 |
|------|------|
| `CIMEntry` 클래스 | ✅ 변경 없음 |
| `ProjectCIM` 클래스 | ✅ 신규 `to_v2()` 추가만 |
| `MultiWorkCIM` 클래스 | ✅ 신규 `upgrade_to_v2()`, `version` 속성 추가만 |
| `init_project`, `record`, `global_weight`, `stats` | ✅ 시그니처 변경 없음 |
| `stats()` 반환 dict | ⚠️ `version` 키 추가 (비파괴적 확장) |

---

## 결과

- 신규 코드는 `create_multi_work_cim()` 팩토리를 통해 v2 API를 기본으로 사용
- 기존 v1 코드는 `MultiWorkCIM` 직접 인스턴스화로 계속 작동
- v1 데이터를 v2로 이관하는 마이그레이션 경로 제공 (`upgrade_to_v2()`)
